1. MCP 是什么

MCP（Model Context Protocol） 是 Anthropic 于 2024 年 11 月发布的开放标准协议，用于 AI Agent 与外部工具/数据源之间的标准化通信。它是 AI Agent 的 "USB-C 接口" — 一次配置，所有兼容 Agent 都能使用。

用户指令 → Agent → MCP Client ↔ MCP Server ↔ 外部工具/API/数据 → 结构化结果返回

核心概念

🔌 MCP Server

提供工具（Tools）的一方。可以是本地进程（stdio）、远程 HTTP 服务，或云服务。每个 Server 暴露一组 Tool 定义（名称、描述、参数 JSON Schema）。

🔗 MCP Client

消费工具的一方。AI Agent（Claude Code、Cursor、Codex、Hermes）内置 MCP Client，在启动时连接所有配置的 Server，将 Tool 列表注入 Agent 上下文。

🛠️ Tool

一个具体的功能函数。Agent 通过 serverName.toolName(params) 直接调用，无需解析文档、无需构造命令。输入输出均为结构化 JSON。

📋 Resource

MCP Server 可暴露的数据源（文件、数据库记录、API 响应）。Agent 可以像读本地文件一样读取远程资源。

为什么需要 MCP？

在 MCP 出现之前，AI Agent 只能通过以下方式与外部工具交互：

# Skill 方式：Agent 读取 Markdown 文档 → 理解怎么用 → 构造 CLI 命令 → 执行 → 解析文本输出 # 问题：耗 Token、容易出错、输出解析不稳定、每个 Agent 需要单独配置
# MCP 方式：Agent 直接调用 tool({params}) → 获取结构化 JSON 结果 # 优势：省 Token（~90%）、类型安全、一次配置所有 Agent 可用
MCP 是开放标准，不由任何单一厂商控制。任何 AI Agent 只要实现 MCP Client 协议即可使用所有 MCP Server。 协议规范：modelcontextprotocol.io

2. MCP vs Skill 深度对比

Skill 和 MCP 是 AI Agent 工具生态的两种范式。理解差异有助于正确选型。

架构对比

维度Skill（技能）MCP（协议）
本质 Markdown 文档 + CLI 脚本 标准化工具协议 + 结构化 Schema
Agent 调用方式 读取 .md 文档 → 理解指令 → 构造命令 → 执行 Bash → 解析文本输出 调用 server.tool(params) → 直接获取 JSON 结果
Token 消耗 高 — 加载完整 Skill 文档（2K-5K tokens），每次调用需上下文 低 — 仅 Tool 定义 (~80 tokens/server)，调用不计入文档
跨 Agent 兼容 Claude Code 专属（Skill 格式为 Claude Code 特有） 所有实现 MCP Client 的 Agent 通用
类型安全 无 — 文本输入输出，需 Agent 自行解析 JSON Schema 强类型约束参数和返回值
错误处理 依赖 Agent 自行判断命令是否成功 标准错误码，Agent 可编程处理
配置方式 放入 ~/.claude/skills/ 目录 写入 Agent 配置文件（JSON/TOML/YAML）
维护成本 每个 Skill 独立维护，格式不统一 统一协议，Server 开发者维护一次
可发现性 需手动搜索、下载、安装 MCP 注册中心 / 市场逐步建设中
版本管理 手动 git pull 更新 远程 Server 自动更新（HTTP 模式）

Token 消耗对比（关键指标）

操作Skill Token 消耗MCP Token 消耗节省
注册工具定义 ~2,500 tokens（加载完整 .md） ~80 tokens（Tool Schema） 97%
调用一次工具 ~200-500 tokens（命令 + 文本解析） ~50-100 tokens（JSON in/out） 75%
10 次调用 ~6,500 tokens ~880 tokens 86%
100 次调用（重度使用） ~52,500 tokens ~5,080 tokens 90%
结论：对于频繁调用的工具（搜索、设计、部署），MCP 的 Token 效率是 Skill 的 10 倍以上。Skill 适合作为一次性参考文档或复杂工作流模板，MCP 适合作为可重复调用的函数接口。

何时用 MCP？何时用 Skill？

✅ 用 MCP

  • API 调用（搜索、数据库、云服务）
  • 需要结构化输入输出
  • 多个 Agent 共用同一工具
  • 频繁调用的操作（>10 次/会话）
  • 有现成 MCP Server

✅ 用 Skill

  • 复杂多步骤工作流模板
  • 需要领域知识文档（非纯 API）
  • Claude Code 专属场景
  • 无 MCP Server 可用的工具
  • 一次性参考手册
不要同时安装 MCP 和同功能 Skill！ 会造成 Agent 同时看到两个工具定义，浪费 Token，且可能因输出不一致导致混淆。优先选 MCP，删除重复 Skill。

3. MCP 传输类型

MCP 支持两种传输方式，适用于不同场景：

stdio 本地进程传输

MCP Server 作为本地子进程运行，通过标准输入/输出（stdin/stdout）通信。Agent 启动时自动拉起进程。

~/.mcp.json — stdio 示例（angular-cli）
{
  "angular-cli": {
    "command": "npx",
    "args": ["-y", "@angular/cli", "mcp"]
  }
}

优点：无网络依赖、延迟低、数据不出本地。
缺点：需本地安装 Node.js/npx，进程常驻内存。

streamable-http 远程 HTTP 传输

MCP Server 作为远程 HTTP 服务运行，Agent 通过 HTTP 请求调用工具。支持 Bearer Token 或 OAuth 认证。

~/.mcp.json — HTTP 示例（anysearch）
{
  "anysearch": {
    "type": "streamable-http",
    "url": "https://api.anysearch.com/mcp",
    "headers": {
      "Authorization": "Bearer YOUR_API_KEY"
    }
  }
}

优点：无需本地安装、自动更新、跨设备可用。
缺点：依赖网络、有延迟、API 需认证。

特性stdiostreamable-http
部署位置本地机器远程服务器
启动方式Agent 自动拉起子进程Agent 发起 HTTP 连接
认证方式无需认证（本地信任）Bearer Token / OAuth 2.1
适用场景开发工具（CLI、编译器）云服务 API（搜索、设计、部署）
资源占用占用本地内存/CPU不占本地资源
更新方式npm update服务端自动更新

OAuth 2.1 认证（Linear 示例）

对于需要用户授权的服务（如 Linear），MCP 支持 原生 OAuth 2.1 流程。Agent 首次连接时自动打开浏览器，用户授权后即可使用。无需手动拷贝 Token。

Agent 启动 → 检测 OAuth Server → 打开浏览器 → 用户授权 → 自动获取 Token → 后续请求自动携带

4. Claude Code MCP 配置

🤖 Claude Code

配置文件：~/.mcp.json（用户级，影响所有项目）

配置格式：JSON

MCP 版本：Streamable HTTP + stdio 双模式

配置文件结构

~/.mcp.json
{
  "mcpServers": {
    "server-name": {
      // ── HTTP 模式 ──
      "type": "streamable-http",
      "url": "https://api.example.com/mcp",
      "headers": {
        "Authorization": "Bearer YOUR_API_KEY"
      }

      // ── 或 stdio 模式 ──
      "command": "npx",
      "args": ["-y", "package-name", "mcp"]
    }
  }
}

CLI 添加命令

bash
# 添加 HTTP MCP Server（用户级，-s user）
claude mcp add -s user --transport http <name> <url>

# 示例：添加 Linear（OAuth 认证）
claude mcp add -s user --transport http linear https://mcp.linear.app/mcp

# 查看已连接的 MCP Server
claude mcp list

# 查看 Server 提供的工具
claude mcp tools <server-name>
Claude Code 专属：~/.mcp.json 是 Claude Code 的全局 MCP 配置。~/.claude/settings.json 不能配置 MCP（会被 JSON Schema 校验拒绝）。

6. Codex (OpenAI) MCP 配置

🟢 Codex (OpenAI)

配置文件：~/.codex/config.toml

配置格式：TOML

前置条件：需开启 rmcp_client = true

配置步骤

1. 开启 MCP 功能：在 config.toml 中添加：

~/.codex/config.toml
[features]
rmcp_client = true

2. 添加 MCP Server：

~/.codex/config.toml（追加内容）
# HTTP 类型（Authorization 放在 http_headers 里）
[mcp_servers.anysearch]
url = "https://api.anysearch.com/mcp"
http_headers = { "Authorization" = "Bearer as_sk_xxx" }

[mcp_servers.figma]
url = "https://mcp.figma.com/mcp"
http_headers = { "Authorization" = "Bearer figd_xxx", "X-Figma-Region" = "us-east-1" }

[mcp_servers.render]
url = "https://mcp.render.com/mcp"
http_headers = { "Authorization" = "Bearer rnd_xxx" }

# OAuth 类型（无需 token）
[mcp_servers.linear]
url = "https://mcp.linear.app/mcp"

# stdio 类型
[mcp_servers.angular-cli]
command = "npx"
args = ["-y", "@angular/cli", "mcp"]
Codex TOML 格式注意事项：
  • streamable_http 不支持 bearer_token，认证 Header 放在 http_headers 里
  • Header 写法：http_headers = { "Authorization" = "Bearer TOKEN" }
  • 必须开启 [features] rmcp_client = true 否则 MCP 不加载

9. Token 效率深度分析

原理

MCP 省 Token 的根本原因：

阶段SkillMCP
工具发现 Agent 读取整个 SKILL.md（含描述、用法、示例、限制说明）→ 2,000-5,000 tokens Agent 获取 Tool JSON Schema（仅名称、描述、参数）→ 60-100 tokens/tool
参数理解 Agent 从 Markdown 文本中推断参数格式 → 可能出错 JSON Schema 严格定义类型、必填/可选、枚举值 → 绝不会出错
执行调用 构造 CLI 命令字符串 → 执行 Bash → 解析 stdout 文本 → 200-500 tokens/次 tool_call({json_params}) → 返回结构化 JSON → 30-80 tokens/次
错误处理 解析错误信息文本 → 重试构造新命令 → 翻倍 Token 消耗 标准错误码 → 直接处理 → 无额外 Token

实际案例：5 个 MCP Server 的 Token 占用

场景Skill 方案MCP 方案节省
加载 5 个工具集~12,500 tokens~400 tokens96.8%
典型会话（30 次工具调用）~19,500 tokens~1,900 tokens90.3%
高强度会话（100 次调用）~52,500 tokens~5,080 tokens90.3%
月均（30 个会话）~585,000 tokens~57,000 tokens90.3%
实际节省远不止 Token 数量。MCP 的结构化输出减少 Agent 推理错误，避免重试浪费。实际场景中 MCP 的 Token 效率是 Skill 的 10-30 倍（考虑错误重试）。

10. 命名空间与冲突处理

MCP 命名空间机制

MCP 使用 serverName.toolName 命名空间，不同 Server 的 Tool 自动隔离，不会冲突。

Tool 调用格式
anysearch.search("Claude 3.5 release date")     ← anysearch 的 search
figma.get_design_context("https://figma.com/...") ← figma 的设计读取
render.create_web_service({...})                  ← render 的部署

即使两个 Server 都有同名的 search Tool，Agent 通过 serverName.search 区分，不会混淆。

MCP + Skill 共存时的冲突

当 MCP 和 Skill 提供相同功能时会有问题：

# ❌ 反模式：同时安装 MCP figma + Skill figma # Agent 看到两套工具描述 → 不确定该用哪个 → 浪费 Token 加载两份定义 # 输出可能不一致（MCP 返回 JSON，Skill 返回文本）
# ✅ 正确做法：仅保留 MCP，删除同功能 Skill # rm -rf ~/.codex/skills/figma* # rm -rf ~/.codex/skills/render-deploy/ # rm -rf ~/.claude/skills/anysearch/
已执行的清理：本项目已删除所有与 MCP 重复的 Skill：
  • ~/.claude/skills/anysearch/ → 已删除（MCP anysearch 替代）
  • ~/.codex/skills/figma*/（7 个目录）→ 已删除（MCP figma 替代）
  • ~/.codex/skills/render-deploy/ → 已删除（MCP render 替代）
  • ~/.codex/skills/linear/ → 已删除（MCP linear 替代）

11. 常见问题 FAQ

MCP 收费吗？

MCP 协议本身完全免费开源。每个 MCP Server 的收费策略由其提供商决定：

  • ✅ anysearch — 免费（1000 次/天，匿名也可用）
  • ✅ angular-cli — 免费（Angular 官方维护）
  • ✅ figma — 免费（需注册 Figma 账号）
  • ✅ render — 免费（需注册 Render 账号）
  • ✅ linear — 免费（需注册 Linear 账号）

为什么 Agent 重启后才加载新的 MCP？

MCP Client 在 Agent 启动时一次性连接所有 Server 并获取 Tool 列表。修改配置文件后需重启 Agent 才能生效。这避免了运行时动态发现的开销。

MCP 连接失败怎么办？

MCP Server 不可用时不会阻断 Agent 启动。Agent 会标记该 Server 为不可用状态，其余 Server 正常使用。常见原因：

  • API Key 过期或无效
  • 网络不通（HTTP Server 需要科学上网？）
  • OAuth Token 未授权（Linear 需首次浏览器授权）
  • stdio Server 依赖未安装（如 npx 不可用）

如何验证 MCP 是否正常工作？

bash
# Claude Code：列出已连接 Server
claude mcp list

# 查看某 Server 的工具列表
claude mcp tools anysearch

# 在 Agent 会话中直接让 Agent 调用试试
"用 anysearch 搜索一下今天的科技新闻"

可以自己写 MCP Server 吗？

可以。MCP 协议是开放的，支持 Python/TypeScript/Go 等多种语言的 SDK。详见 MCP 官方文档。一个最简单的 MCP Server 只需要几十行代码。

Skill 会被淘汰吗？

不会完全淘汰，但角色会转变：

  • MCP → 标准化 API 调用（未来主流）
  • Skill → 领域知识手册、复杂工作流模板、教育参考
  • 两者互补，不是替代关系
