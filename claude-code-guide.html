1. Claude Code 是什么

Claude Code 是 Anthropic 出品的终端 AI 编程助手，不是一个代码补全插件，而是一个具有完整工具访问权限的自主 Agent。

用户输入指令 → 理解意图 → 搜索代码库 → 读取文件 → 分析与推理 → 执行 Bash 命令 → 编辑文件 → 验证结果 → 回复用户

它能做什么？

🔍 代码搜索

用 Glob 匹配文件名、Grep 正则搜内容，ripgrep 级别的搜索速度

✏️ 文件编辑

精确字符串替换（Edit）、整体覆写（Write）、Jupyter Notebook 编辑

⚡ 命令执行

运行测试、构建项目、安装依赖、Git 操作，一切 Bash 命令

🤖 多 Agent 协作

启动子 Agent 并行处理任务，Workflow 编排大规模自动化

🌐 网络能力

Web 搜索最新信息、抓取网页内容、访问在线文档

📋 项目管理

创建和跟踪 TODO 列表，制定实施计划，管理任务依赖

4. 核心心智模型：如何用好 Claude Code

法则一：说目标（What），不说步骤（How）

Agent 自己决定工具调用顺序。你描述期望结果，它规划执行路径。

坏："用 grep 搜所有 auth 文件，然后把它们全部读一遍，分析清楚中间件逻辑……"
好："重构 auth 模块，把 token 校验逻辑从路由层抽到独立 middleware"
法则二：给上下文，一次性给全

文件路径、行号、错误信息、环境版本——全部在第一轮给出。Agent 追问 = 浪费 token 和回合。

坏："报错了"
好："app.py:42 TypeError: 'NoneType' object is not subscriptable，Python 3.12，FastAPI 项目"
法则三：多个需求放一回合

相关需求合在一起提交，Agent 并行处理。分开提交 → 每次都重载上下文 → 浪费 token。

坏：第 1 轮 "加个 API"，第 2 轮 "写个测试"，第 3 轮 "更新文档"
好："加 POST /api/user，写单元测试 + 集成测试，更新 README"
法则四：复杂任务先 Plan

涉及多文件、架构变更、大重构时，先让 Agent 出完整方案（Enter Plan Mode），确认后再执行。

好："先出方案：把支付模块从单体拆成微服务。Plan mode。"
法则五：信任 Agent 的工具选择

不要说"用 grep 搜"、"用 sed 改"。Agent 有 Glob/Grep/Edit/Bash 全套工具，它知道什么场景用什么。你只需描述结果。

法则六：用 CLAUDE.md 做项目记忆

Claude Code 每次对话开头自动读取 CLAUDE.md。把架构约定、代码风格、常用命令写进去，Agent 自动理解你的项目。

好：运行 /init 生成模板 → 填入项目规范 → Agent 每轮自动读取
你的角色不是指挥者，是 Product Manager。你定义 What & Why，Claude Code 负责 How。和好工程师协作一样——给清楚背景和验收标准，放手让它做。

5. Token 优化圣经：让每分钱花在刀刃上

Claude Code 按 token 计费。输入的 token + 输出的 token 都要钱。以下策略从最有效到最激进排列：

5.1 最高 ROI：Caveman Mode

输入 /caveman，Claude 会用极简风格回复，输出 token 降低约 75%。信息密度反而更高。

前："好的，我已经理解了您的需求。让我先搜索相关代码，然后进行分析，最后给出建议……"
后："done. 改 3 files. tests pass."
快速迭代时必用。当你有大量任务要串行处理时效果最显著。

5.2 减少回合数 = 减少上下文重复加载

策略说明预估节省
一次说清全部需求相关需求合并成一个 prompt30-50%
给完整上下文文件路径、行号、错误信息一次给全20-30%
用 Plan Mode 先确认方案避免走弯路 → 删掉重来40-60%
精确描述而非宽泛"改 UserService 的 login 方法" > "改登录"20-40%

5.3 善用 /compact

对话长了以后，输入 /compact 让系统自动压缩上下文为摘要。摘要会保留关键信息，丢掉已完成的冗余细节。相当于「整理聊天记录」。

压缩有损：有时关键细节会被摘要掉。压缩后检查 Agent 是否还能正确理解你的意图。如果跑偏，说明需要重新描述。

5.4 让 Agent 读文件，而非你贴内容

坏：把 500 行代码直接贴到聊天里 → 两者都消耗 input token
好：说 "看 src/auth.py:30-80" → Agent 自己读，只消耗它的一次 Read 调用
你贴内容 = 你的 input token 也被计费。给路径让 Agent 自己读 = 更便宜。

5.5 避免在 Agent 工作时插入干扰

Agent 在执行任务时，你突然说"顺便改个别的"→ 上下文混乱 → Agent 重读代码 → 浪费 token。

✅ 等当前任务结束，或说 "停，新需求：……" 让它重新组织计划。

5.6 复杂任务用 Workflow 而非对话驱动

如果你有大量文件要审计/迁移/生成，让 Agent 用 Workflow 工具编排多个子 Agent 并行处理，比你在对话里一个文件一个文件地让它改高效得多。Workflow 并行调度比对话轮询带宽大 10 倍以上。

5.7 Token 消耗自查清单

检查项不 OKOK
是否一次性给了上下文Agent 追问 2+ 次Agent 直接开始干活
是否有不相关的大段内容贴了 500+ 行无关代码给了精确路径
是否一个会话做多个任务改了 Auth 又改 UI 又改 DB一个会话一个完整功能
是否启用了 caveman modeAgent 回复大段文字简洁短小
是否先用 plan 确认了方向执行到一半发现方向错Plan 确认后一步到位

6. 写出好内容的 Prompt 工程

同样是 Claude Code，同样的模型，prompt 的质量决定了输出的质量。以下是针对 Claude Code 场景的专用技巧：

6.1 指定输出格式

坏："写个 API 文档"
好："用 OpenAPI 3.0 格式写 API 文档，包含 200/400/500 响应示例，Markdown 文件输出到 docs/api.md"

6.2 给约束边界

场景指令
不改某个文件"别碰 config.py"
限定技术栈"用原生 JS，不引入框架"
代码风格"遵循现有代码风格，不做无关重构"
测试覆盖"先写测试再写实现，branch coverage ≥ 90%"
架构约束"新增的模块放 src/services/ 下，不改 API 签名"
只分析不动手"只出分析报告，不要改代码"

6.3 用角色设定提升质量

坏："review 这段代码"
好："作为安全审计专家，review 这段代码，重点检查 OWASP Top 10 漏洞"
角色设定的本质是帮助 Agent 选择合适的「脑内知识库」。说"作为 React 性能专家"比"优化性能"更精准，因为 Agent 会激活 React 特定的优化模式。

6.4 示例驱动：描述输入 → 输出

坏："重构这段屎山代码"
好："下面的函数太长了，拆成 3 个职责单一的小函数，每个不超过 20 行。不改 public API，现有测试必须通过"

6.5 善用 Claude 的 Structred Output 能力

当你需要 Agent 返回特定结构（JSON 配置、代码模板等），直接描述 schema：

好："生成一个 JSON 配置文件，包含：name(string)、version(semver)、routes(array of {path, method, handler})。用 camelCase。"

6.6 迭代式提问：从模糊到精确

第一轮：宽泛描述 → 看输出 → 第二轮：加约束 → 看输出 → 第三轮：微调细节

不要期待一轮完美。把 Agent 当成 junior developer 来指导——先让 ta 做个大概，你审完再给具体修改意见。三轮之后通常质量远超一次性「完美 prompt」。

避免微管理：每行代码都让 Agent 按你的意思改 = 浪费 token。描述可验证的验收标准（"所有测试必须通过"），而不是逐行指导。你的时间比 token 贵。

7. 典型工作流实战

7.1 加新功能

描述功能 + 给约束 → Agent 自动读代码 → 实现 → 跑测试 → 你验收
大功能先 Plan。Agent 会读完所有相关文件再动手，不会盲改。

7.2 修 Bug

复现步骤 + 错误堆栈 → /diagnose → Agent 定位 → 修复 → 验证
用 /diagnose 触发系统化排障流程，比直接说"修一下"效果好得多。

7.3 代码审查

/review → Agent 读 diff → 输出结构化审查意见 → 选择是否应用建议

7.4 TDD 开发

描述功能 → /tdd → Agent 写测试（Red） → 写实现（Green） → 重构（Refactor） → 循环

7.5 大规模重构

描述目标 + 不变约束 → Plan Mode 出方案 → 确认 → 逐模块实施 → 跑全量测试 → /verify 验证

7.6 定时任务

/loop 5m /verify → 每 5 分钟自动检查部署状态

8. 常见反模式

❌ 一次给超模糊需求

"帮我做个网站" → Agent 要猜框架、设计、功能、部署方式，大概率翻车。

✅ "用 React + Tailwind 做个个人博客，有文章列表和 Markdown 详情页，部署到 GitHub Pages"

❌ Agent 工作中插入新需求

Agent 正在编辑多个文件，你突然说"顺便把 XX 也改下"→ 上下文污染 → 两个任务都可能出错。

✅ 等当前任务结束，或明确说 "停止当前任务，新需求：……"

❌ 用 Agent 做不需要推理的机械操作

"把这 100 个 JSON 文件格式化" → 直接用 Prettier，别用 AI。AI 贵且可能出错。

✅ Agent 适合需要理解代码、做决策、跨文件关联分析的任务。纯机械操作用脚本。

❌ 让 Agent 自我循环修复超过 3 次

Agent 陷入"改 → 报错 → 再改 → 再错"的死亡循环 → 烧大量 token 但毫无进展。

✅ 3 次失败后喊停，自己看一下。"停，我来看看"。你的 30 秒审视可能省下 100k token。

❌ 在一个超长会话里做所有事

上下文膨胀 → 早期指令被压缩 → Agent "忘记"最初目标 → 输出质量断崖下降。

✅ 一个会话 = 一个完整功能。完成就用 /compact 或开启新会话。用 /handoff 交接上下文。

一句话总结

说清目标、给够上下文、一回合说完、复杂先计划、信任工具调用。
