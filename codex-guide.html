1. 核心心智模型

Codex 不是搜索引擎，不是代码补全工具——它是一个有工具、能执行命令、可编辑文件的自主 Agent。

你发指令 → 理解意图 → 搜索代码 → 读文件 → 分析 → 执行命令 → 编辑文件 → 验证 → 回报结果

关键点：Agent 自己决定工具调用顺序。你不需要告诉它每一步怎么做，你只需要说清楚目标。

2. 五条黄金法则

法则一：说目标，不说步骤

Agent 需要知道你要什么，不需要知道怎么做。

坏："用 grep 搜 auth 相关代码，然后把找到的文件都读一遍，分析中间件逻辑……"
好："重构 auth 模块，把 token 校验逻辑从路由层抽到独立 middleware"
法则二：给上下文，别让我猜

文件路径、行号、错误信息、环境信息——一次性给全，Agent 不需要追问。

坏："报错了，帮我看看"
好："app.py:42 TypeError: 'NoneType' object is not subscriptable。Python 3.12, FastAPI 项目"
法则三：一回合说完

多个相关需求放一起，Agent 能并行处理。分开说 → 每次重新加载上下文 → 浪费 token。

坏：第一轮 "加个 API"，第二轮 "补测试"，第三轮 "改下文档"
好："加 /api/user 接口，写单元测试，更新 README"
法则四：复杂任务先计划

涉及多文件、大重构、架构变更时，说 plan mode 或 先出方案。Agent 会先出完整计划，你确认后再执行，避免走错路。

法则五：让 Agent 自己决定工具

不要说 "用 grep 搜" 或 "用 sed 改"。Agent 有完整的工具集，它知道该用什么。你只需描述结果。

2.1 Agent 的"手感"——高质量协作模式

Codex 自带工程品味：它会自动尊重现有代码风格、避免过度抽象、在合适的尺度插入注释。理解这些特质，你能更快进入 flow。

Codex 会做什么你无需担心
读取代码库找已有模式不用教它用哪个框架
并行搜索、读文件、执行命令不用等它一步步回复
添加注释只在代码不够自解释时不会塞满无意义注释
编辑保持最小范围不会顺手重构无关代码
报错时主动搜索、分析、修复不用你 debug
工作时持续给状态更新不用猜测它在干嘛

2.2 Agent 的记忆机制——为什么它会「忘」

Agent 的记忆不是你想的那样。它不是硬盘，更像一块 黑板——能写的东西有限，写满了就得擦掉旧的。

记忆怎么工作的

记忆类型存什么存活时间容量
对话上下文本次会话的全部对话、代码、文件内容仅当前会话有限（token 上限）
文件系统项目代码、文档、AGENTS.md、ADR永久无上限
Skills / Plugins固化在 .codex/skills/ 的专项能力跨会话持久无上限

关键在于：对话上下文是最活跃但最脆弱的记忆。Agent 每轮决策都依赖它，但它有严格的容量限制。超限时系统会自动压缩旧内容成摘要——压缩过程中细节会丢失。

记忆混乱的典型场景

场景一：上下文溢出

你在一轮对话里给了太多文件、太长代码、太多不相关的指令。Agent 的上下文窗口被填满，早期指令被压缩或丢失，导致「忘了你最初要干嘛」。

场景二：任务干扰

Agent 正在做任务 A，你突然插入任务 B 的指令。上下文里同时存在两个目标，Agent 可能混淆优先级，同时处理两个任务或干脆搞错目标。

场景三：幻觉递归

Agent 在某一步做出了错误判断，这个错误被写入上下文，然后 Agent 基于这个错误继续推理——错误自我强化，越来越离谱。

场景四：跨会话断裂

你在上次会话让 Agent 改了一个模块，下次开新会话让它继续优化同一个模块。但它完全不记得上次干了什么——因为没有文件系统的变更记录可读。

怎么避免记忆问题

策略具体做法解决什么问题
一个会话一件事搞完一个完整功能再开下一个。别边改 A 模块边让 Agent 修 B 模块的 bug任务干扰
写项目记忆文件创建 AGENTS.md（架构/约定）和 CONTEXT.md（领域术语/决策记录）。Agent 每轮自动读取跨会话断裂
用 ADR 记录决策每次架构决策写一条 ADR 放 docs/adr/。新会话读了就理解历史决策跨会话断裂
用 Handoff 交接复杂任务跨多轮时，说 handoff 生成结构化交接文档。下一轮 Agent 读文档续上上下文溢出 + 跨会话断裂
精简指令别贴大段无关代码、别同时提 5 个需求。需要文件时给路径让 Agent 自己读上下文溢出
用 Caveman Mode说 caveman mode，Agent 回复极短省 token ~75%，留给上下文的可用空间更多上下文溢出
及时纠正发现 Agent 逻辑跑偏，立刻纠正，别等它自己「想通」。说 停，方向错了：[正确方向]幻觉递归
定期检查上下文对话长了就回头看 Agent 前面的理解对不对。不对就重新描述目标上下文溢出 + 幻觉递归

记忆机制的优缺点

优点缺点
即开即用：新会话不需要初始化配置无跨会话记忆：每轮是「新人」
上下文内推理强：窗口内的信息能精确关联容量有限：超出 token 上限就丢信息
文件系统补偿：读代码 = 读项目历史压缩失真：自动摘要会丢关键细节
Skills 可迁移：写好一次到处用需人工维护：AGENTS.md/ADR 要人写
隐私安全：会话结束记忆消失长对话成本高：token 消耗随对话长度线性增长
终极建议：把 Agent 当成一个 极其聪明但每次醒来都失忆的工程师。你给它留的 AGENTS.md 就是它的笔记本，代码就是它的记忆。依赖文件系统而非对话上下文做长期记忆。

2.3 Codex 内置工具引擎 —— 它怎么干活

Codex 没有斜杠命令。它的「手」是这些底层工具，每次对话中自动组合调用。理解这些工具 = 理解它的能力边界。

文件与系统操作

工具能力典型场景
shell_command在 PowerShell 沙箱中执行任意命令：文件搜索 (rg/ls)、构建、Git 操作、Python 脚本"跑一下测试" "装依赖" "git diff"
apply_patch (add_file)创建新文件，填写完整内容"建一个 app.py"
apply_patch (update_file)精确打补丁：只改目标行，不动周围代码"在 login 函数里加一行日志"
apply_patch (replace_file)全文件替换覆盖"重写这个配置文件"
apply_patch (delete_file)删除文件"删掉废弃的 old_schema.sql"
apply_patch (batch)单次批处理多个文件操作，原子级修改"批量改 10 个 API handler 的签名"
Codex 默认用 rg (ripgrep) 搜索代码，比 grep 快一个数量级。给路径让它读文件，别手动贴代码到聊天里。

信息获取

工具能力典型场景
web_search搜索网页获取实时信息：最新文档、API 变更、方案调研"查一下 React 19 新特性" "搜这个报错的方案"
view_image读取并分析本地图片：截图诊断、设计稿重建、图表解读"看看这张 UI 截图有什么问题"

任务规划与目标跟踪

工具能力典型场景
update_plan创建并管理分步任务清单，标记 pending / in_progress / completed多文件重构时自动拆分步骤并追踪进度
create_goal设定工作目标 + 可选 token 预算限 token 的任务场景
get_goal查看当前目标、已用/剩余 token、耗时检查是否超预算
update_goal标记目标完成 (complete) 或因阻塞终止 (blocked)任务结束/卡住三轮以上
request_user_inputPlan 模式下发起选择题确认方向"用 PostgreSQL 还是 MySQL？"
工具调用哲学

Codex 自己决定用哪个工具、什么顺序、是否并行。你只需说目标。它默认并行搜索和读取文件，单次可同时发起多个独立操作。

2.6 两种工作模式

Default Mode（默认）

收到目标后直接动手：搜索→读代码→分析→改文件→验证。适合明确的需求、修 bug、加小功能。

Plan Mode（计划）

先出方案确认方向再执行。可调用 request_user_input 做选择题。适合多文件重构、架构变更。

Codex 的工程价值观

无论哪种模式，Codex 自带三层判断：

价值行为表现
Clarity 清晰显式解释推理过程，不让你猜
Pragmatism 务实专注实际效果、推进进度，不过度设计
Rigor 严谨发现漏洞或弱假设时礼貌指出，不做表面功夫

3. 典型工作流

3.1 加新功能

描述功能 → Agent 读代码 → 出方案 → 你确认 → 实现 → 测试 → 跑验证
大功能先用 plan mode。Codex 会读完相关文件再动手，不会盲改。

3.2 修 Bug

描述现象 + 错误信息 → Agent 定位代码 → 修复 → 验证
给堆栈跟踪、复现步骤。Codex 自带 diagnosis skill（diagnose），说 "diagnose this" 会启动系统化排查。

3.3 重构

说目标 → Agent 分析依赖 → 逐模块实施 → 跑测试
说清楚重构边界和约束。"拆 auth 模块但不改 API 签名"。

3.4 测试驱动开发 (TDD)

说 "tdd" → Agent 写测试 → 写实现 → 重构 → 循环
Codex 内置 tdd skill。说 "用 TDD 加这个功能"，它会自动走 red-green-refactor 循环。

Codex 有大量内置 skill，每个都是专项能力包。常见：

Skill作用触发方式

4.2 多 Agent 协作

复杂项目可以串多个 Agent：第一个出方案，第二个实现，第三个 review。用 handoff skill 做上下文传递。

4.3 Token 预算意识

Agent 按 token 计费。精简指令 = 省钱。用 caveman mode 降低回复开销。大任务先 plan mode 确认方向再执行，避免浪费。

4.4 给 Agent 设定约束

约束场景指令
不改某个文件"别碰 config.py"
限定技术栈"用原生 JS 不用框架"
测试优先"先写测试再实现"
保持简洁"caveman mode"
只分析不动手"只出方案" 或 plan mode

5. 常见反模式

❌ 一次给一个超模糊的需求

"帮我做个网站" → Agent 要猜框架、设计、功能、部署……必然翻车。

✅ 给 scope："用 Vue 做个个人博客，有文章列表和详情页，Markdown 渲染"

❌ 在 Agent 工作过程中插入新需求

Agent 正在编辑文件，你突然说 "顺便把 XX 也改下" → 上下文混乱。

✅ 等当前任务结束，或说 "停，新需求：……" 让它重新组织

❌ 用 Agent 做不需要推理的事

"把这份 JSON 格式化" → 你不如用 Prettier。Agent 贵。

✅ 把 Agent 用在需要理解代码、做决策的复杂任务上

❌ 让 Agent 自己验证太久

Agent 陷入循环修复 → 反复尝试 → 浪费大量 token。

✅ 失败 3 次后暂停，人工审查。说 "停，我来看看"

6. 度量你用得对不对

指标好差
追问次数0-13+
Agent 搜索次数3-1050+（目标不清）
每轮文件改动数1-50（没做事）或 20+（盲目铺开）
错误重试次数1-25+（需求有矛盾）
从指令到完成的轮次1-38+

回头翻翻 Agent 对话日志。如果你的指令经常触发 50+ 次搜索、8+ 轮对话——不是 Agent 不行，是你的指令需要改进。

7. 总结

Codex 是一把好刀。好刀没法替代你思考，但能把你的思考执行得飞快。

一句话
说清目标、给够上下文、一回合说完、复杂先计划、信任工具调用。
你负责 What & Why，Agent 负责 How。

8. 按职业角色的最佳实践

8.1 如果你是前端开发

场景最佳指令模板
从设计稿到组件"用 Tailwind + React 实现这个卡片组件，三个状态：default / hover / loading"
调 CSS 布局"这个 flex 布局在 Safari 下错位了，帮我改"（附带截图）
加动画"用 GSAP 给这个 section 加滚动触发动画，触发点在视口 80% 处"
写页面"做一个仪表盘页面，用现有设计系统的 Card/Table/Chart 组件，数据 mock 就行"
做组件库"给 Button 加 variant='outline'、loading 状态和 disabled 样式"

前端关键：先给 Agent 看现有组件和设计 token。它自动匹配风格，不会造新的按钮系统。复杂交互优先用 GSAP（Codex 内置完整的 GSAP 技能链）。

8.2 如果你是后端开发

场景最佳指令模板
加 API"加 POST /api/orders，body 字段：user_id, items[]。校验库存，返回 order_id。已有 DB model 在 models/order.py"
写 middleware"在 auth.py 加 rate limiting middleware，Redis 存储，60req/min per IP"
数据库迁移"给 users 表加 last_login_at 和 login_count，写 Alembic migration"
重构"把 payment 模块从单体 service 拆成 payment_gateway + payment_log 两个模块，不改对外接口"
写 integration test"给这个 CRUD 接口写集成测试，覆盖正常流 + 边界（空 body、超长字段、重复 key）"

后端关键：说清楚数据模型、边界条件、SQL/ORM 偏好。Codex 默认用项目已有框架。批量操作时让它用 apply_patch_batch 一次改多文件。

8.3 如果你是 AI / ML / 数据分析

场景最佳指令模板
数据处理"读 data/raw.csv，清洗空值和异常值，输出 data/clean.csv。用 pandas。"
模型训练"用 PyTorch 搭一个 LSTM 做时间序列预测，输入 30 天预测 7 天。train.py 和 model.py 分开。"
可视化"用 matplotlib 画训练 loss 曲线和验证集准确率，两张图并排，保存 PNG"
Agent 开发"基于 OpenAI function calling 写一个 Python agent loop，含 tool registry 和 retry 机制"
调 prompt"这个 prompt 输出格式不稳定，帮我加 structured output 约束改用 JSON mode"

AI/ML 关键：明确数据文件路径和输出格式。Codex 能跑 Python 脚本，让它直接执行并验证结果。

8.4 如果你是 DevOps / SRE

场景最佳指令模板
写 Dockerfile"给这个 Python 项目写多阶段 Dockerfile，base python:3.12-slim，用 uv 装依赖"
CI/CD"写 GitHub Actions workflow：lint → test → build Docker → push ECR"
K8s"给这个 app 写 deployment + service + ingress yaml，namespace 用 'prod'"
脚本"写 bash 脚本定时清理 7 天前的日志文件，加 dry-run 模式"
监控"给 FastAPI 项目加 Prometheus metrics endpoint，统计请求延迟和状态码"

DevOps 关键：基础设施细节给全（镜像、版本、端口、环境变量）。Codex 会匹配项目已有配置风格。

8.5 如果你是产品经理 / 技术写作

10. Agent 技术学习路径（详细版）

如果你不只是想用好 Agent，还想自己开发 Agent 系统——以下是从零到一的完整 roadmap，每个阶段都包含 学什么、怎么学、免费资源、动手练习。

L1 基础 → L2 跑起来 → L3 理解框架 → L4 工程化 → L5 前沿

L1 基础 · 2-4 周

学什么：Python（或 TypeScript）熟练到能独立写脚本、调包；理解 HTTP/REST 请求与响应、JSON 序列化；LLM 核心概念——token、context window、system prompt、temperature、tool calling 的请求格式。

怎么学：不要只看教程，边看边写。Python 的话先把官方 Tutorial 过一遍（2-3 天），然后直接去 LeetCode 刷 20 道 Easy 题熟悉语法。LLM 概念不要死记，直接注册 OpenAI 账号，在 Playground 里调 temperature、改 system prompt、用不同的 model，看输出怎么变——10 分钟比读一小时文档有效。

免费资源链接推荐理由
Python 官方教程docs.python.org最权威，中文版完整
TypeScript Handbooktypescriptlang.org官方手册，示例清晰
DeepLearning.AI 短课deeplearning.aiChatGPT Prompt Engineering 免费课，1.5 小时上手
OpenAI Playgroundplatform.openai.com不写一行代码就能调模型参数，理解 temperature/top_p/max_tokens
MDN HTTP 指南developer.mozilla.orgHTTP 请求/响应、状态码、Header，前端/后端都必看
LeetCodeleetcode.cnEasy 题刷 20 道，语法就够用了
动手练习：用 Python 调 OpenAI Chat Completions API，写一个命令行问答脚本。要求：支持多轮对话（用 messages 数组维护上下文），传 temperature 参数，出错时 retry。

L2 动手 · 1-2 周

学什么：从零实现一个最简单的 Agent Loop——while 循环 + LLM 调用 + 工具执行 + 结果拼回 prompt。再手写一个 RAG Pipeline：文档切 chunk → embedding → 存入向量数据库 → 检索 → 拼接 prompt → 生成回答。

怎么学：先手写，别用框架。Agent loop 本质就四步：发 prompt → 解析响应（判断是调工具还是最终回答）→ 执行工具 → 结果塞回上下文，循环。RAG 也一样，自己调 embedding API + chromadb/chroma 搭一遍，比看十篇博客都管用。

免费资源链接推荐理由

Chroma 向量数据库trychroma.comPython 本地运行，5 行代码搭向量检索，RAG 入门首选
ollamaollama.com本地跑开源模型（Llama/Mistral/Qwen），免费无限调
动手练习：① 写一个能调用天气 API + 计算器的 Agent（至少两个 tool）；② 写一个本地文档问答系统——丢进去 10 篇 Markdown 文档，能检索并回答内容相关问题。

L3 理解框架 · 2-4 周

学什么：深入理解成熟的 Agent 框架——LangChain 的 AgentExecutor、Memory、Chain 机制；AutoGPT 的任务分解和自循环；CrewAI 的多 Agent 角色协作。对比 ReAct、Plan-Execute、Multi-Agent 三种范式的适用场景和优劣。

怎么学：核心方法是「读源码 + 自己实现」。LangChain 的 AgentExecutor 就几百行核心代码，逐行读懂它怎么 parse tool call、怎么 decide next action。然后不用 LangChain，用你在 L2 写的裸 Agent loop 复现相同效果。三种范式各写一个 demo 对比，你会发现差异非常大。

免费资源链接推荐理由

动手练习：① 读 LangChain AgentExecutor 源码并写注释博客；② 分别用 ReAct 和 Plan-Execute 实现同一个任务（例如「分析一个 GitHub repo 并生成 report」），对比两者的 token 消耗和执行质量。

L4 工程化 · 持续

学什么：Tool 设计最佳实践（description 怎么写才能让 LLM 正确调用、错误信息如何向上传递、幂等性如何保证）；Eval 评估体系（怎么量化 Agent 表现——准确率、延迟、token 消耗、tool call 正确率）；Prompt Engineering 进阶（few-shot、chain-of-thought、structured output）；Guardrails 安全护栏（输入/输出校验、PII 脱敏、越狱防护）。

怎么学：工程化不是学出来的，是「踩坑踩出来的」。把你 L3 的项目丢进真实场景，用户用着用着就出问题——tool 选错了、prompt 被绕过、延迟太高——每个问题都是一个学习点。建立自己的 eval set：收集 50-100 个真实用例，每次改动后跑一遍，这才是工程化的核心。

动手练习：① 给你 L3 的 Agent 建一个 eval set（至少 20 条），记录每次改 prompt 后的 accuracy；② 接入 Guardrails AI 做输出校验，确保 Agent 不会返回格式错误或越界内容；③ 用 Braintrust 或 Langfuse 搭一个可观测面板，能看到每次调用的 token、延迟、tool call 详情。

L5 前沿 · 持续

学什么：跟踪前沿论文——ReAct、Tree of Thoughts、Reflexion、SWE-bench 系列。理解 Agent 协议标准——MCP（Model Context Protocol）让 Agent 统一访问工具和数据源，A2A（Agent-to-Agent）让多个 Agent 互相发现和协作。最终目标：自己设计并实现一个完整的 Agent 系统。

怎么学：论文不是用来「读」的，是用来「复现」的。选一篇你感兴趣的（推荐从 ReAct 开始），读懂算法后用自己的代码实现核心逻辑。协议类的最好上手试——给 MCP 写一个 server，或者搭两个 Agent 走 A2A 互相调用。读论文的节奏：每周 1-2 篇，精读而非泛读。
